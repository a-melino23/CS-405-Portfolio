## Course Overview

CS-405 focused on secure coding practices and the importance of building security into each stage of software development. The course covered topics such as secure design, threat awareness, exception handling, unit testing, static analysis, encryption, authentication, authorization, accounting, and security policy creation.

A major takeaway from this course was that security should be treated as an ongoing concern during development. Secure coding is not just about fixing problems after they appear. It is about reducing risk early by writing safer code, testing expected and unexpected behavior, using automation where appropriate, and documenting standards that developers can follow consistently.

## Final Project

The final project for this course centered on creating a secure coding policy and presentation for Green Pace. The policy was designed to help guide a development team toward more consistent and secure software practices.

The project included recommendations for:

- Secure coding standards
- Encryption policies
- Authentication, authorization, and accounting controls
- Static analysis and automated security testing
- Unit testing
- Exception handling
- DevSecOps automation
- Secure development practices across the software lifecycle
- Risk identification and prioritization

The final project helped connect the technical work from earlier assignments to a larger professional security policy. It also showed how secure coding practices can be applied at both the code level and the organizational level.

## Reflection

This course reinforced that secure coding is a process, not a single step. I learned that vulnerabilities can come from many different areas, including weak input validation, poor exception handling, unsafe memory use, weak authentication, missing encryption, and inconsistent testing.

The work in this repository shows my growth in thinking about security earlier in the development process. Instead of only focusing on whether code runs, I worked on considering how it could fail, how it could be misused, and how it could be improved to reduce risk.
